Current Stack

Desktop-first, Tauri-native, still vanilla on the frontend.

Odyssey is no longer a browser app backed by a local Node server. The current app is a Tauri 2 desktop shell that serves static HTML from public/, persists data through the Tauri FS plugin, and splits gameplay orchestration across public/creation.js, public/chat.js, and public/saves.js. When updating docs or features, assume native desktop first.

Frontend Static HTML pages, shared CSS, shared theme engine, large vanilla JS modules.
Native Layer Tauri 2 window shell, Rust bootstrap, bundled dialog/fs/shell plugins.
Persistence App data lives under Tauri's app data directory, not inside the repo.

Overview

The old devdocs drifted because they still described a web-server shape that no longer exists. The current source of truth is straightforward:

Native window

src-tauri/tauri.conf.json points frontendDist at ../public and launches the app on Welcome.html.

Single shared frontend

The menu, game screen, settings, presets, and wiki are all plain HTML pages living in public/.

Bridge-first storage

Save files and preset files are created via window.tauriBridge, not a local HTTP API.

No docs-only design language

Developer docs should inherit the same dark palette and default fonts the game uses so they feel like part of Odyssey.

Mental model: Odyssey is a static frontend bundled into a native shell. The browser APIs are still used for UI work, but the app behaves like a desktop client with direct local persistence.

Runtime Map

Boot sequence

  1. Tauri launches a frameless native window defined in src-tauri/tauri.conf.json.
  2. The first page loaded is public/Welcome.html.
  3. Shared scripts like theme.js, titlebar.js, menu-music.js, and tauri-bridge.js wire up the app shell behavior.
  4. The user moves through the menu into setup and gameplay, while settings and data are read from localStorage plus files in the app data directory.

All screens are shipped as static files and share the same theme and storage conventions.

Reusable character preset management backed by local JSON files.

Shared assumptions

  • Pages lean on global state and shared browser APIs rather than a component framework.
  • theme.js is responsible for CSS variables and font loading.
  • titlebar.js matters because the Tauri window is frameless.
  • When you change storage or gameplay contracts, the impact usually spans multiple HTML pages plus creation.js, chat.js, and saves.js.

Tauri Shell

The Rust side is intentionally thin. The app does not expose a large custom command layer yet; instead it uses Tauri's built-in plugins plus the browser-side bridge.

Important config facts

  • frontendDist points at ../public, so the HTML files in public/ are the packaged frontend.
  • The main window starts hidden, is undecorated, and is configured to open at Welcome.html.
  • withGlobalTauri: true is enabled, which is why window.__TAURI__ is available to the frontend.
  • src-tauri/src/main.rs is just the bootstrap entry that delegates to app_lib::run().
{
  "build": { "frontendDist": "../public" },
  "app": {
    "withGlobalTauri": true,
    "windows": [
      {
        "decorations": false,
        "maximized": true,
        "url": "Welcome.html"
      }
    ]
  }
}

Storage Model

There are two persistence layers in play: lightweight config in localStorage, and structured save/preset data on disk through Tauri FS.

App data directory layout

public/tauri-bridge.js resolves path.appDataDir(), appends odyssey, then manages the following folders:

      npc-ledger.json

Bridge responsibilities

  • listPresets, savePreset, deletePreset
  • listGames, loadGame, saveNewGame, deleteGame, updateGame
  • downloadImage, getBaseImage, updateImage
  • exportGame and import/export dialog flows through the Tauri dialog plugin
Contributor rule: if you are documenting or changing persistence, document the Tauri bridge first. Old references to /api/save-new-game and similar local server endpoints are legacy fallback paths, not the primary architecture.

Setup Flow

The old monolithic gameplay module is now split by responsibility: creation.js owns new game setup, chat.js owns prompts and in-game chat, and saves.js owns autosaves and chat history persistence.

New game phases

  1. World: real-world start or custom universe, including optional preset data from WORLD_PRESETS.
  2. Player: character sheet, attributes, relationships, inventory, and preset loading.
  3. Scenario: opening situation written by the player.
  4. Summary: AI-generated campaign summary and confirmation step.
  5. Launch: initial save written through window.tauriBridge.saveNewGame(), optional portrait download, then transition into gameplay.

Schema-driven responses

The main game loop expects structured JSON described by gameOutputSchema. Required top-level keys are:

{
  "time": { "hour": 0, "minute": 0, "period": "AM", "dayOfWeek": "", "day": 1, "month": 1, "year": 1, "era": "", "calendarType": "" },
  "textoutput": "",
  "inventory_changes": [],
  "location_changes": [],
  "npc_changes": [],
  "stats": { "health": 100, "money": 0, "hunger": 0, "thirst": 0, "energy": 100 }
}

Provider Flow

Odyssey talks to OpenAI-compatible chat endpoints, but the implementation has provider-specific branches. This is one of the places old docs aged out fastest.

Current request model

  • buildFetchPayload() assembles the chat payload for text generation.
  • Presence and frequency penalties are skipped for xAI.
  • LM Studio gets a response_format.type = "json_schema" block tailored to its expectations.
  • Generic OpenAI-compatible mode may need prompt-enforced JSON or repair logic if the backend does not fully support structured outputs.

OpenAI-compatible notes worth preserving

Base URL shape

Store custom provider URLs at the API root such as https://host.example/v1, not the full /chat/completions endpoint.

Manual model IDs

Some providers support chat completions but not a useful /models endpoint, so settings include a manual model path.

Structured output drift

Local or third-party servers often fail on response_format. The app surfaces targeted user guidance when that happens.

Repair pass

For OpenAI-compatible providers, malformed JSON can be repaired through a follow-up parse/repair path before the turn is rejected.

Game Loop

The gameplay turn loop is still frontend-managed. The app builds context, calls the model, parses structured output, updates state, and persists the changed files through the bridge.

Per-turn flow

  1. Load world, player, gamestate, ledgers, summary, and chat history into memory.
  2. Build a game system prompt with current state and optional lore augmentation.
  3. Send a structured chat completion request with gameOutputSchema.
  4. Parse or repair the result into narration, codex updates, stat changes, and inventory changes.
  5. Render the markdown output and persist updated state via window.tauriBridge.updateGame().
  6. If enabled, kick off auto-image generation for the latest scene.

What gets updated after a turn

  • gamestate.json
  • chat_history.json
  • npc-ledger.json
  • locationsledger.json
  • Optional current scene image metadata

Images and TTS

Odyssey now has two notable auxiliary systems that the old docs barely covered: image generation and voice playback.

Image generation

  • Portrait and scene image work is coordinated through creation.js for initial setup and chat.js for runtime updates.
  • The app can use a separate image provider from the text provider.
  • LM Studio is explicitly blocked for image generation with a user-facing error message.
  • Base character images and latest scene images are stored alongside the save.

TTS providers

  • Kokoro / OpenAI-compatible speech endpoint
  • OpenAI speech
  • Google Cloud TTS
  • xAI TTS
Why this matters for docs: settings, saved keys, and runtime wiring now span text, image, and audio providers. Any contributor doc that only talks about text generation is incomplete.

Theme and Fonts

The docs page should behave like the app, not like a random marketing microsite. Odyssey's visual system lives in public/theme.js and public/style.css.

Default game typography

  • --font-primary: default heading font is Cinzel
  • --font-secondary: default body font is Merriweather

Dark theme source of truth

The dark preset defines the palette for the current docs refresh: black base backgrounds, translucent dark surfaces, white headings, muted gray body text, and subtle white borders. This page now applies those same tokens directly in its own CSS so it stays visually aligned with Odyssey without mutating the app's saved theme state.

:root {
  --bg-main: #000000;
  --bg-secondary: rgba(15, 15, 15, 0.6);
  --text-main: #a1a1aa;
  --text-heading: #ffffff;
  --font-primary: "Cinzel", serif;
  --font-secondary: "Merriweather", serif;
}

Contributor guidance

  • Use existing CSS variables before inventing new colors.
  • Prefer var(--font-primary) and var(--font-secondary) over page-specific font stacks.
  • If you add a new docs component, it should still look correct under the shared dark token set.

Dev Workflow

Commands

npm install
npm run dev
npm run build

What these do now

  • npm run dev starts the Tauri desktop app in development mode.
  • npm run build produces native bundles under src-tauri/target/release/bundle/.
  • There is no standalone npm start local web server in the current package scripts.
Docs maintenance rule: whenever scripts, plugin usage, or save paths change, update this page and the top-level README.md together so contributor guidance does not fork.

Guardrails

When changing docs

  • Document the Tauri path first, and only mention legacy fallback code when it still affects behavior.
  • Match the actual filenames and script names in the repo.
  • Call out whether behavior is powered by local files, localStorage, or a provider API.

When changing code

  • If you alter save file names or structure, update tauri-bridge.js docs and any import/export notes.
  • If you touch theme tokens or font defaults, check this page and the player-facing wiki for visual drift.
  • If you add a new provider branch, document its base URL expectations and structured-output quirks.
